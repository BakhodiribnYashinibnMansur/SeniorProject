# Go fmt — Professional / Production Level

## 1. Overview

In production code, `fmt` is the package you reach for **less and
less**. Most of its appearances are in tests, errors, and CLI tools;
service logs go to `slog`/`zap`/`zerolog`, hot-path conversions go
to `strconv`, and structured output goes to `encoding/json`. What
remains is exactly where `fmt` shines: error wrapping, `Stringer`
methods on domain types, and one-shot formatting.

This leaf documents the rules production teams settle on after a few
on-call shifts: where `fmt` is allowed, what `vet`/`staticcheck`
catch, the `Stringer` and `Formatter` patterns from real OSS, and
the trade-offs between `Printf`-line logging and structured logging.

---

## 2. The "Where is fmt Allowed" Convention

Most teams arrive at a rule like this:

| Place | fmt allowed? | Reason |
|-------|--------------|--------|
| Domain `String()` methods | Yes | This is what `Stringer` is for |
| Error wrapping (`Errorf` with `%w`) | Yes | Canonical idiom |
| `tests/` and `examples/` | Yes | Output is for humans |
| CLI tools | Yes | Output is for humans |
| HTTP handler bodies | Sometimes | Prefer `json.Encoder` for APIs |
| Service logs (long-running) | **No** | Use `slog`/`zap`/`zerolog` |
| Hot-path string building | No | Use `Builder` + `strconv` |
| User-facing translations | No | Use `golang.org/x/text/message` |

`golangci-lint` ships rules to enforce this; see Section 5.

## 3. Stringer in the Standard Library

`fmt.Stringer` is implemented across the stdlib. A non-exhaustive
list:

| Type | File | Notes |
|------|------|-------|
| `time.Time` | `src/time/format.go` | RFC 3339 default |
| `time.Duration` | `src/time/time.go` | `1h2m3s` form |
| `time.Weekday` | generated by `stringer` | `Sunday`, `Monday`, ... |
| `time.Month` | generated by `stringer` | `January`, ... |
| `net.IP` | `src/net/ip.go` | dotted/colon forms |
| `net.IPNet` | `src/net/ip.go` | CIDR form |
| `*os.File` | `src/os/file.go` | filename |
| `bytes.Buffer` | `src/bytes/buffer.go` | string view |
| `regexp.Regexp` | `src/regexp/regexp.go` | source pattern |
| `reflect.Type` | `src/reflect/type.go` | type name |
| `errors.errorString` | `src/errors/errors.go` | error message (via `error`, not `Stringer`) |

Every one of these is what makes `fmt.Println(time.Now())` work
without ceremony.

## 4. Real OSS Examples

### 4.1 cockroachdb/cockroach — `tree.Expr` Formatter

```go
// pkg/sql/sem/tree/format.go (simplified)
type Expr interface {
    Format(ctx *FmtCtx)
}

func (e *BinaryExpr) Format(ctx *FmtCtx) {
    ctx.Format(e.Left)
    ctx.WriteByte(' ')
    ctx.WriteString(e.Operator.String())
    ctx.WriteByte(' ')
    ctx.Format(e.Right)
}
```

CockroachDB does not use `fmt.Formatter` directly; it wraps a
`bytes.Buffer` in `FmtCtx` and exposes `Format(ctx)` so that
formatting can carry context (redacted values, alias maps,
locality). The lesson: `fmt.Formatter` is convenient but inflexible.
For a SQL engine you write your own `Format` protocol.

### 4.2 prometheus/common — `model.LabelValue`

```go
// vendor/github.com/prometheus/common/model/labels.go
type LabelValue string

func (v LabelValue) String() string {
    return string(v)
}
```

A direct `Stringer` on a string newtype. The reason: `LabelValue` is
unicode-safe but not necessarily UTF-8 valid; `String()` is the
canonical conversion. Used everywhere in alertmanager output.

### 4.3 grpc-go — `status.Status`

```go
// google.golang.org/grpc/internal/status/status.go
func (s *Status) String() string {
    return fmt.Sprintf("rpc error: code = %s desc = %s",
        codes.Code(s.s.GetCode()), s.s.GetMessage())
}
```

`codes.Code` itself implements `Stringer` (generated by `stringer`).
This is the format you see in every gRPC-related panic and log
line.

### 4.4 etcd-io/raft — `raftpb.MessageType`

```go
// raft/raftpb/raft.pb.go (proto-generated, then `stringer -type`)
func (x MessageType) String() string {
    return proto.EnumName(MessageType_name, int32(x))
}
```

Proto enums get a free `Stringer` from the `proto` runtime; etcd's
log lines (e.g. `received MsgVote from 5...`) all feed through it.

### 4.5 kubernetes — apimachinery enums

```go
// staging/src/k8s.io/apimachinery/pkg/types/types.go
//go:generate stringer -type=PatchType
type PatchType string

const (
    JSONPatchType           PatchType = "application/json-patch+json"
    MergePatchType          PatchType = "application/merge-patch+json"
    StrategicMergePatchType PatchType = "application/strategic-merge-patch+json"
    ApplyPatchType          PatchType = "application/apply-patch+yaml"
)
```

`stringer` is the convention for enums across the Kubernetes
codebase.

### 4.6 hashicorp/go-multierror — Format

Older `multierror` predates `errors.Join`:

```go
// github.com/hashicorp/go-multierror/multierror.go
func (e *Error) Error() string {
    fn := ListFormatFunc
    if e.ErrorFormat != nil {
        fn = e.ErrorFormat
    }
    return fn(e.Errors)
}
```

A configurable formatter. Modern code uses `errors.Join` from the
stdlib instead.

### 4.7 pkg/errors — Stack Trace via Format

```go
// github.com/pkg/errors/errors.go
func (w *withStack) Format(s fmt.State, verb rune) {
    switch verb {
    case 'v':
        if s.Flag('+') {
            fmt.Fprintf(s, "%+v", w.Cause())
            w.stack.Format(s, verb)
            return
        }
        fallthrough
    case 's':
        io.WriteString(s, w.Error())
    case 'q':
        fmt.Fprintf(s, "%q", w.Error())
    }
}
```

`%+v` to expose the stack trace, plain `%s`/`%v` for the message.
This pattern is reproduced in `cockroachdb/errors`, `pingcap/errors`,
and roughly every Go ORM's error type.

## 5. Lint Rules

### 5.1 vet's printf analyzer

Built into `go vet`. Flags:

- `Printf format %d has arg "x" of wrong type string`
- `Printf format %s reads arg #2, but call has 1 arg`
- `Printf format with non-constant string` (when enabled)
- `%w used outside of fmt.Errorf`

It runs on every `go test`. Treat its warnings as errors in CI.

### 5.2 staticcheck

| Check | Description |
|-------|-------------|
| `SA1006` | `Printf` with dynamic format string and no format args. Likely a verb-injection bug. |
| `SA9006` | `Printf`-style call with format-arg type mismatch (vet on steroids). |
| `S1025` | Unnecessary `Sprintf` (e.g. `Sprintf("%s", s)` → `s`). |
| `SA1010` | `regexp.MustCompile` of `Sprintf` result — likely a bug. |

### 5.3 revive

| Rule | Description |
|------|-------------|
| `error-strings` | Error strings should not be capitalized or end with punctuation (so they compose under `%w`). |
| `unnecessary-format` | `Errorf` with no verbs → use `errors.New`. |

### 5.4 golangci-lint preset

A typical `.golangci.yml`:

```yaml
linters:
  enable:
    - govet           # printf analyzer
    - staticcheck     # SA1006, SA9006, S1025, SA1010
    - revive          # error-strings, unnecessary-format
    - errorlint       # %w usage, errors.Is over ==
linters-settings:
  govet:
    enable:
      - printf
  errorlint:
    errorf: true
    errorf-multi: true
    asserts: true
    comparison: true
```

`errorlint` deserves special mention: it catches `Errorf("...: %v", err)`
and recommends `%w`, plus `err == sentinel` and recommends
`errors.Is`.

## 6. fmt vs slog/zap/zerolog

### 6.1 The Trade-off

| Aspect | `fmt.Println`/`Printf` | `slog`/`zap`/`zerolog` |
|--------|------------------------|------------------------|
| Output | Free-form text | Structured (JSON/logfmt) |
| Allocations | ~2/call | 0 with handlers like `slog.JSONHandler` (Go 1.22+) or zerolog |
| Filtering | Grep | Severity + key matching |
| Aggregation | Manual | Native key indexing |
| Latency | ~100ns/call | ~50ns/call (zerolog), ~80ns (slog) |
| Schema | Implicit | Explicit |

For a service running 10k log lines/sec, the per-call cost matters
and the structure matters even more — splunk/datadog charge by
ingested bytes, and structured logs compress and index better.

### 6.2 When to keep fmt

- CLI tools: `fmt.Println("Loaded 12 records")` is just right.
- Test failure messages: `t.Errorf("want %d, got %d", w, g)`.
- Error wrapping: `fmt.Errorf("step: %w", err)`.

### 6.3 When to switch

Anywhere a log line is consumed by a machine — log aggregator,
alerting, dashboarding — switch to structured logging.

```go
// Before
fmt.Printf("user=%d action=%s latency=%v\n", uid, action, dur)

// After
slog.Info("action complete",
    "user", uid,
    "action", action,
    "latency", dur,
)
```

## 7. The %w Discipline

A common production rule:

1. **Always** wrap with `%w`, never `%v`, when returning an error
   the caller might inspect.
2. **Don't** wrap a wrapped error twice with the same prefix; the
   chain fragment becomes redundant.
3. Use sentinels for boundaries:
   ```go
   var ErrNotFound = errors.New("not found")
   return fmt.Errorf("user %d: %w", id, ErrNotFound)
   ```
4. Use typed errors when callers branch on `errors.As`:
   ```go
   type DBError struct{ Op string; Err error }
   func (e *DBError) Error() string { return fmt.Sprintf("db.%s: %v", e.Op, e.Err) }
   func (e *DBError) Unwrap() error { return e.Err }
   ```
5. Multiple `%w` (Go 1.20+) for joining; or `errors.Join`.

## 8. Stringer for Logs and Dashboards

When a domain enum hits a log line or a dashboard, you want a stable
string, not an integer:

```go
type Severity int

const (
    SevDebug Severity = iota
    SevInfo
    SevWarn
    SevError
)

//go:generate stringer -type=Severity -trimprefix=Sev
```

After `go generate`, `Severity.String()` produces `Debug`, `Info`,
`Warn`, `Error`. Logs are searchable; dashboards are readable.

The `-trimprefix` flag drops the `Sev` so the string is `Info` and
not `SevInfo`.

## 9. Observability and Errors

Modern observability stacks (Honeycomb, OpenTelemetry, Datadog) tag
spans with errors. They typically capture `err.Error()` as the
message. Two implications:

1. Your `Error()` method must be safe to call repeatedly — no
   side-effects.
2. Your `Error()` should include enough context to be searchable
   (`load /etc/x: open /etc/x: permission denied` is searchable;
   `bad config` is not).

`fmt.Errorf("op %s: %w", op, err)` produces searchable messages
naturally because `op` shows up in the prefix.

## 10. Format Strings in Internationalisation

`fmt` is **not** an i18n package. Verbs are positional and not
designed for translation reordering. Use
[`golang.org/x/text/message`](https://pkg.go.dev/golang.org/x/text/message)
for proper i18n:

```go
import "golang.org/x/text/message"
import "golang.org/x/text/language"

p := message.NewPrinter(language.German)
p.Printf("There are %d cats\n", 3) // "Es gibt 3 Katzen"
```

`message.Printer` understands plurals, genders, and reorderable
arguments. `fmt` does not.

## 11. Real-World Anti-Patterns

### 11.1 fmt.Sprintf in HTTP Handlers

```go
// Bad — JSON in a string
fmt.Fprintf(w, `{"user":"%s","age":%d}`, u.Name, u.Age)

// Good
json.NewEncoder(w).Encode(u)
```

The `Sprintf` form is a JSON-escaping bug waiting to happen. Use
`encoding/json`.

### 11.2 fmt.Errorf with %v for Wrapping

```go
// Bad — chain broken
return fmt.Errorf("load: %v", err)

// Good
return fmt.Errorf("load: %w", err)
```

`errorlint` flags this.

### 11.3 fmt.Println in Hot Loops

```go
// Bad — 1M allocations/sec
for _, x := range items {
    fmt.Println("processing", x.ID)
}

// Good — slog with batched output
slog.Info("processing", "id", x.ID)
// or move the log out of the loop.
```

### 11.4 fmt.Sprintf for SQL

```go
// CRITICAL — SQL injection
q := fmt.Sprintf("SELECT * FROM users WHERE id=%s", req.ID)

// Good
db.Query("SELECT * FROM users WHERE id=$1", req.ID)
```

`gosec` and `sqlclosecheck` catch this.

### 11.5 Custom Format That Loses Information

```go
func (e *DBError) Format(s fmt.State, verb rune) {
    fmt.Fprintf(s, "%s", e.Op) // ← drops e.Err
}
```

A custom `Format` must handle every realistic verb, including
`%+v` for verbose output and `%v` for default.

## 12. Production Checklist

- [ ] Format strings are constants.
- [ ] All wrap-style errors use `%w`.
- [ ] No user input in `Printf` first arg.
- [ ] No `Sprintf`-built SQL.
- [ ] Hot-path conversions use `strconv`.
- [ ] Service logs go to `slog`/`zap`/`zerolog`.
- [ ] Domain types implement `Stringer` where logged.
- [ ] Enums use `stringer` codegen.
- [ ] `errorlint` is in CI.
- [ ] `vet` printf warnings break the build.

## 13. Common Mistakes

| Mistake | Fix |
|---------|-----|
| `fmt.Println` in service code | `slog.Info` |
| `Errorf` with `%v` | `%w` |
| Sprintf-built SQL | parameterised queries |
| Stringer recurses on self | alias type or explicit fields |
| Pointer-only Stringer | value receiver |
| Custom Format dropping info | handle all common verbs |

## 14. Common Misconceptions

**Misconception 1**: "fmt is fine for service logs."
**Truth**: Free-form text is hard to index. Structured logging wins
at scale.

**Misconception 2**: "fmt.Sprintf is faster than json.Marshal."
**Truth**: For one field, yes. For a struct of N fields, json is
roughly the same and produces parsable output.

**Misconception 3**: "%+v is enough for debugging."
**Truth**: It is for primitives. For nested structs, json with
indent is more readable.

**Misconception 4**: "vet catches all printf bugs."
**Truth**: It catches mismatches on **literal** format strings. If
the format is built at runtime, vet is silent.

**Misconception 5**: "errorlint is opinionated overkill."
**Truth**: The `%w`-discipline rule alone has caught dozens of
bugs in our incident postmortems. Keep it on.

## 15. Tricky Points

1. `errorlint` flags non-`%w` wraps; `staticcheck` SA1006 flags
   format-as-variable. Both belong in CI.
2. `vet` does not see custom `Printf`-likes; document `printfuncs`.
3. `slog` is Go 1.21+; older codebases may still use third-party
   loggers. The `%w` discipline applies regardless.
4. `Format` methods that don't honour width/precision are common;
   document the supported flags.
5. `Errorf` with multiple `%w` is Go 1.20+; older codebases must use
   single-`%w` and wrap manually.

## 16. Test (Production-Style)

```go
package userdb_test

import (
    "errors"
    "testing"
)

func TestNotFoundUnwraps(t *testing.T) {
    _, err := userdb.Get(ctx, "missing")
    if !errors.Is(err, userdb.ErrNotFound) {
        t.Fatalf("expected ErrNotFound, got %v", err)
    }
}
```

This style assumes the implementation wraps with `%w` and exposes
sentinels. Anywhere it doesn't, this test would fail.

## 17. Tricky Questions

**Q1**: Why does `errorlint` flag `fmt.Errorf("...: %v", err)`?
**A**: Because the wrapped error is no longer recoverable via
`errors.Is` — the chain is broken.

**Q2**: How do you write a `Stringer` for a redacted field?
**A**:
```go
func (c Credentials) String() string {
    return fmt.Sprintf("Credentials{User:%q, Pass:[redacted]}", c.User)
}
```

**Q3**: When is `fmt.Sprintf` justified in a hot path?
**A**: When the call rate is bounded and clarity beats nanoseconds.
A retry-once error message: yes. A per-request log line: no.

## 18. Cheat Sheet

```go
// Wrap (production discipline)
return fmt.Errorf("op %s: %w", op, err)

// Stringer for enum
//go:generate stringer -type=Status -trimprefix=Status

// Format method respecting +v
func (e *T) Format(s fmt.State, verb rune) {
    switch verb {
    case 'v':
        if s.Flag('+') {
            fmt.Fprintf(s, "%s\n  cause: %+v", e.Op, e.Err)
            return
        }
        fallthrough
    case 's':
        fmt.Fprint(s, e.Error())
    }
}

// errorlint config
linters: { enable: [errorlint] }
linters-settings:
  errorlint:
    errorf: true
```

## 20. Summary

`fmt` survives in production for three reasons: domain `Stringer`
methods, error wrapping with `%w`, and CLI/test output. Everywhere
else — service logs, JSON responses, hot-path conversions — better
packages exist. Lock down the discipline with `vet`, `staticcheck`,
and `errorlint`, and use `stringer` codegen for enums. The result is
code that produces searchable errors, parsable logs, and readable
debug output without surprising allocation profiles.
